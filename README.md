# Healthcare Claims Analytics (dbt + Snowflake)

A dbt + Snowflake project where I modeled synthetic healthcare claims data into a layered warehouse - staging, intermediate, and marts - with tests and CI.

Built as a portfolio project to get hands-on with analytics engineering on a modern cloud stack. Uses [Synthea](https://synthea.mitre.org/) synthetic patient data (free, HIPAA-free, generated by MITRE).

## What's in it

- 8 raw tables loaded into Snowflake (~278K rows)
- 19 dbt models across staging, intermediate, and marts layers
- Star schema: 4 dimensions (patients, providers, payers, date) + 2 facts (claims, encounters)
- 4 analytics marts answering payer-side business questions
- 57 data tests running on every build
- GitHub Actions CI workflow
- Auto-generated dbt docs with lineage

## Architecture

![architecture](docs/architecture.svg)

| Layer | Materialization | What it does |
|---|---|---|
| RAW | table | Synthea CSVs loaded as-is, one table per file |
| staging (`stg_*`) | view | Rename to snake_case, cast types, simple derivations |
| intermediate (`int_*`) | ephemeral / view | Joins and business logic prep |
| marts/core (`dim_*`, `fct_*`) | table | Star schema dims and facts |
| marts/analytics (`mart_*`) | table | Pre-aggregated tables for analytics |

One note on Azure: in a real production setup, the CSVs would land in Azure Blob Storage and get ingested via Snowpipe from an external stage. I used Snowflake's internal UI loader to keep setup simple for the portfolio.

## The 4 analytics marts

1. **`mart_member_cost_summary`** - per-patient cost summary with decile ranking, used to spot high-cost members
2. **`mart_provider_performance`** - claims volume, unique patients, avg cost, outstanding rate per provider
3. **`mart_payer_analytics`** - per-payer enrollment, encounter mix, coverage ratios
4. **`mart_chronic_condition_costs`** - cost burden aggregated by chronic condition cohort (diabetes, CHF, COPD, etc.)

## Running it

Needs Snowflake (free trial works), Python 3.9+, git.

```
git clone https://github.com/SoumyaShahh/claims-analytics-dbt-snowflake.git
cd claims-analytics-dbt-snowflake
python -m venv venv
.\venv\Scripts\Activate.ps1
pip install dbt-core==1.8.7 dbt-snowflake==1.8.4
```

Run the two scripts in `snowflake/` as ACCOUNTADMIN to set up the database, warehouse, role, user. Then drop Synthea CSVs into `data/` and load them using Snowflake's UI loader.

Configure `~/.dbt/profiles.yml` with your Snowflake account details, then:

```
cd claims_analytics
dbt deps
dbt build
```

Should see 19 models built and 57 tests passing in ~30 seconds.

For docs:

```
dbt docs generate
dbt docs serve
```

## Tests

57 tests run on every build - unique, not_null, relationships (FK integrity between facts and dims), and accepted_values on enum columns like gender.

![dbt build success](docs/screenshots/dbt-build-success.png)

## CI

`.github/workflows/dbt_ci.yml` runs `dbt build` on every PR. Credentials come from GitHub Secrets so nothing sensitive is in the repo.

## Notes

- Synthetic data, so the numbers are illustrative not clinical
- One bug I ran into: `int_patient_chronic_conditions` as ephemeral broke downstream because Snowflake's column inference was dropping the `has_*` flags. Kept it as a view instead.
- `START` and `STOP` are reserved in Snowflake so they stay quoted in the encounters and conditions models
